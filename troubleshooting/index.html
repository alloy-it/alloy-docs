    Troubleshooting¶

    Solutions to the most common problems when using alloy-host and alloy-provisioner.

    alloy-host installation¶

    alloy-host: command not found¶

    The binary is not in your PATH.

    macOS / Linux: Move it to /usr/local/bin/ or add the install directory to PATH:

    sudo mv alloy-host /usr/local/bin/

    Windows: Add the folder containing alloy-host.exe to your user PATH in System Properties → Environment Variables.

    check-health reports Vagrant or VirtualBox not found¶

    Most common cause: the tool was not installed, or it's installed in a non-standard location.

    1. Verify the tool is installed by running it directly:
    vagrant --version
VBoxManage --version
    1. If they run but alloy-host doesn't find them, set the path explicitly:
    alloy-host config set vagrant-path /full/path/to/vagrant
alloy-host config set vbox-manage-path /full/path/to/VBoxManage
    1. On macOS with Homebrew, VBoxManage is typically at: /Applications/VirtualBox.app/Contents/MacOS/VBoxManage

    alloy-host up and provisioning¶

    The VM starts but provisioning fails¶

    The provisioner runs inside the guest and needs internet access. Check:

    1. Internet access — the guest VM must reach the Alloy registry and external package servers. If you're behind a proxy or firewall, whitelist those hosts.

    2. Disk space — the first run can download and install several GB of tools. Ensure at least 20 GB free.

    3. Re-run provisioning — run alloy-host provision <name> to re-run provisioning without destroying the VM. The provisioner is idempotent and will resume where it left off.

    vagrant up times out or hangs on "Waiting for machine to boot"¶

    Usually a VirtualBox issue on the host.

    1. Open the VirtualBox GUI and confirm it launches without errors.

    2. Load VirtualBox kernel extensions:

    3. macOS: System Settings → Privacy & Security — approve the VirtualBox kernel extension if prompted.

    4. Linux: sudo modprobe vboxdrv.

    5. Restart your machine and try again.

    alloy-host up fails with "directory already exists"¶

    You already initialised an environment with that name. Use a different name, or to wipe and recreate:

    alloy-host destroy my-vm --all
alloy-host init my-vm --blueprint community/arm-none-eabi
alloy-host up my-vm

    macOS — Apple Silicon (M1/M2/M3)¶

    VirtualBox support for Apple Silicon is experimental. If alloy-host up fails:

    1. Use VirtualBox 7.1 or later for the best ARM Mac support.
    2. Approve the VirtualBox system extension in System Settings → Privacy & Security after installation.
    3. If problems continue, try the WSL2 backend on a Windows machine, or native Linux.

    SSH¶

    alloy-host ssh says "VM is not running"¶

    Start the VM first:

    alloy-host up my-vm
alloy-host ssh my-vm

    SSH connection refused or times out¶

    1. Wait for alloy-host up to finish before running ssh.
    2. Restart the VM: alloy-host stop my-vm then alloy-host up my-vm.

    USB devices¶

    See USB & Target Boards for setup. If devices don’t appear or attach fails:

    • Device not in usb list: Unplug and replug; confirm the host sees it (macOS: System Information → USB; Linux: lsusb).
    • Linux attachment fails: Add your user to vboxusers: sudo usermod -aG vboxusers $USER, then log out and back in.
    • WSL2: Install usbipd (winget install usbipd), open a new PowerShell; if needed, run attach from an elevated PowerShell.

    Environment registry¶

    alloy-host list shows an environment that no longer exists on disk¶

    If you removed the environment directory manually, the registry entry in ~/.alloy-it/vm/vms.yaml is stale. You can edit that file and remove the entry.

    Getting more help¶

    Run commands with --help for usage:

    alloy-host --help
alloy-host up --help
alloy-provisioner -version

    For provisioner issues (native or inside a VM), see Installing on Linux (Native) and Provisioner CLI.
